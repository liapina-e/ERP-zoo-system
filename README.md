# Консольное приложение, реализующее базовую ERP-подсистему Московского зоопарка

Программа позволяет:
- учитывать животных и предметы, стоящие на балансе зоопарка;
- принимать животных с проверкой ветеринарной клиники;
- учитывать потребление корма;
- определять животных, пригодных для контактного зоопарка;
- выводить отчёты в консоль.
- взаимодействовать с пользователем через меню в консоли.

Проект разработан с соблюдением принципов **SOLID** и использованием **DI-контейнера**.

**Примененные принципы SOLID:**

**S**ingle Responsibility Principle  
Каждый класс в проекте выполняет чётко определённую роль:

- Animal и его наследники (Herbo, Predator, Monkey, Rabbit, Tiger, Wolf) описывают модель живых существ и их уникальные свойства (потребление еды, доброта и т.д.).
- Thing и его наследники (Table, Computer) описывают материальные предметы инвентаря.
- Zoo управляет коллекциями животных и предметов, добавлением новых сущностей и их хранением.
- VeterinaryClinic отвечает только за проверку здоровья новых животных.
- ZooPrinter занимается исключительно формированием и выводом отчётов.
- Program и MethodsForConsole — за пользовательское взаимодействие и DI-конфигурацию.

**O**pen/Closed Principle  
Система легко расширяется за счёт использования базовых абстракций:

- Добавление новых видов животных не требует изменения существующего кода — достаточно унаследоваться от Herbo или Predator и определить нужные свойства.
- Аналогично, можно добавить новые виды предметов, унаследовавшись от Thing (например, Table, Computer и т.д.).
- Методы, которые работают с базовыми типами (Animal, Thing), автоматически поддерживают новые подклассы.

То есть код открыт для расширения, но закрыт для модификации.

**L**iskov Substitution Principle  
Любой объект подкласса может быть использован там, где ожидается объект базового класса, без нарушения логики программы:

- Экземпляры Rabbit, Tiger и т.д. могут быть добавлены в зоопарк как Animal.
- Table и Computer используются как Thing при инвентаризации.
- Методы вывода и подсчёта корма работают с базовым типом Animal, не зная о конкретных реализациях.

**I**nterface Segregation Principle  
Интерфейсы разделены по смыслу:

- IAlive описывает живые сущности и содержит свойство Food.
- IInventory описывает сущности, подлежащие инвентаризации (вещи и животные) и содержит свойство Number.  
Животные реализуют оба интерфейса, а вещи — только IInventory.

**D**ependency Inversion Principle  

- Классы `Zoo` и `VeterinaryClinic` теперь зависят от интерфейсов `IZoo` и `IVeterinaryClinic`, а не от конкретных реализаций `Zoo` и `VeterinaryClinic`.
- Объекты создаются и связываются с помощью встроенного DI-контейнера (Microsoft.Extensions.DependencyInjection) в Program.cs.
- Это позволяет подменять зависимости на другие реализации (например, заменить VeterinaryClinic на фейковую для тестов) без изменения основного кода.

**Тестирование**
Проект покрыт модульными тестами (папка kpo_hw1.Tests) с использованием xUnit.  
Покрытие кода ≥ 74 %.  
  
Проверяются ключевые сценарии:
- Добавление животных и вещей.
- Поведение ветеринарной клиники.
- Подсчёт еды и вывод животных для контактного зоопарка
- Методы для работы с консолью
- Корректность вывода данных
- Корректное внедрение зависимостей и связывание компонентов через интерфейсы (проверяется с использованием DI-контейнера)

**Требования**

- .NET 8 SDK или выше
- Rider / Visual Studio / любая IDE с поддержкой .NET

**Запуск программы**
1. Клонировать репозиторий или распаковать архив.
2. Открыть папку kpo_hw1 в Rider или VS.
3. Собрать проект (Build).
4. Запустить через терминал или IDE.
5. Следовать подсказкам в консоли для добавления животных и вывода отчётов.

**Примечания**  
- Архитектура спроектирована с учётом будущего расширения: планируется добавление учёта людей и дополнительных видов инвентаря.
- Все сущности реализуют строгие интерфейсы и следуют принципам SOLID.
- Программа устойчива к ошибкам и корректно обрабатывает ввод пользователя.
